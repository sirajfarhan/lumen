# Skill Density

Use this reference when a skill is substantial or broad. Use it for judgment-heavy skills and for skills that keep failing in similar ways. Use it when the user asks about density. Use it for thinness and resilience work. Use it for reinforcement-loop work.

Skill density means regulatory sufficiency per unit of future-agent attention. A dense skill gives a fresh agent the minimum useful regulator for a task family. The agent can tell what matters. It can vary its response when the request changes. It can produce the work and repair the work. It can stop or adapt when the task leaves scope.

Thinness is not brevity. A short skill can be dense when every sentence changes the path from recognition into return. A long skill can be thin when it lists instructions without a working model. It is also thin when one example carries too much weight, the validator rewards a proxy, or boundary cases depend on hidden agent knowledge.

## Density Card

Use the card as a design review. Write short answers in prose. Do not turn it into a mechanical checklist or a summed score.

Target task family names the request family and the disturbances that matter. A dense answer makes scope and variation visible. A thin answer names a broad outcome without saying what changes across cases.

Minimum internal model names what the agent must internalize before the procedure can travel. It should show what to notice. It should say what must remain true. It should make choice visible. It should show how failure appears and where repair begins. A dense answer explains why the method works. A thin answer mostly gives steps.

Attention-leak control belongs inside the minimum internal model. The agent should know the accepted question under discussion and the side frames that are out of lane. Without that boundary, extra intelligence can make the skill worse by finding attractive adjacent work.

Generation path explains how a fresh agent moves from trigger to first acceptable output. A dense answer supports production, not only recognition. A thin answer has a stronger validator than generator.

Variety coverage names the contrast the task family needs. A dense answer shows when to switch strategy. A thin answer gives near-duplicate examples or one canonical template. When examples teach a broad skill, they should vary the carrier while preserving the same governing perspective. The agent should see the invariant survive a changed medium and a changed reader burden.

Validator target names what the validator actually tests and what it intentionally ignores. A dense answer tests the intended capability at the right level. A thin answer rewards surface shape. It may reward expected phrases. It may reward formatting or length.

Repair map links failures to a changed next decision. The repair may change policy. It may change generation. It may change evidence or fallback. A thin answer says to polish or retry. It may only say to be more careful.

Boundary and fallback names how the skill should behave when the request leaves its reliable region. A dense answer knows when to ask or assume. It knows when to retrieve more context. It knows when to lower confidence. It knows when to defer or stop. A thin answer looks universal until it suddenly fails.

Behavioral evidence names what happened under fresh use at a fixed attention budget. A dense answer shows perturbation handling, repair gain, and honest out-of-scope behavior. A thin answer has only local file checks or canonical demos.

## Coupling Scan

After the first review, scan for mismatches. A weak trigger fails to activate the right model. A strong invariant with weak examples preserves judgment but not fluent execution. A strong validator with a weak repair map creates repeated surface fixes. A rich reference with a weak generation path behaves like a library instead of a skill. A missing boundary makes the package look capable until the first surprise.

## Healthy Loops

A healthy loop starts with model-led generation. Validation then checks behavior that matters to the capability, not an easy proxy. Repair changes the likely cause of failure rather than post-processing the answer. The learning residue is a better discrimination, boundary, or route choice.

False loops are common. A proxy loop rewards formatting or vocabulary until the skill optimizes the wrong signal. A template loop makes examples into fixed answer shapes. A perspective loop is healthier: the example names what viewpoint or boundary must survive, then tests that relation on a different surface. A self-sealing loop explains away failure without revising the governing assumption. An overfit loop improves benchmark cases while becoming brittle on nearby requests.

An attention-leak loop is another false loop. The skill notices a possible concern, treats it as responsibility, and validates that it handled the concern even though the concern never answered the user. Repair by moving the relevance boundary upstream so the concern must earn its place before generation.

## Placement

Put density support where it earns attention. Keep the main skill file focused on recognition and route choice. It should also keep public validation and return visible. Move conditional detail into a reference only when it changes judgment for a real branch. Put exact repeatable checks in scripts. Keep behavior runs and source research outside the package.

When density is weak, do not add a length target. Do not add an example quota. Do not add a validator quota or total density score. Repair the coupling that failed. Narrow scope when variety is too wide. Add contrast when transfer is weak. Add generation logic when the agent can recognize quality but cannot produce it. Add boundary handling when edge cases create confident misuse.
